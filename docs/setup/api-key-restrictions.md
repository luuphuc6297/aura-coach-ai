**Status:** P0 production blocker. Complete every section below before tag-and-ship.

The current keys in this repo were committed to git history while we
were iterating, so they must be assumed compromised — anyone who
cloned the repo has them. Treat each key as "public" until rotated.

## 1. Gemini API key

**Where it's used:** `.env` → `GEMINI_API_KEY` → consumed by
`google_generative_ai` package via `flutter_dotenv`.

**Compromised key:** `AIzaSyBsED7R3HF9dGIPQPbQz8F-v3NpkeFPKsc`

### Rotation steps

1. Go to https://aistudio.google.com/apikey
2. Find the leaked key, click **Delete**.
3. Click **Create API key** → choose project `aura-coach-ai` → copy the new value.
4. Open https://console.cloud.google.com/apis/credentials → switch to project `aura-coach-ai` → click the new key.
5. Under **API restrictions** → select "Restrict key" → enable **only**:
   - Generative Language API
6. Under **Application restrictions** → select **Android apps**:
   - Add package `com.auracoach.aura_coach_ai` + your release SHA-1 fingerprint (from `keytool -list -v -keystore ~/.android/debug.keystore` for debug, or the upload keystore SHA-1 for release).
7. Click **Add another** → select **iOS apps**:
   - Add bundle id `com.auracoach.auraCoachAi`.
8. Save. Replace the value in your local `.env`:
   ```
   GEMINI_API_KEY=<new-key>
   ```
9. Run `flutter clean && flutter run` to verify the key works.

### After rotation

- Force-push a commit that strips the old key from git history if the repo is public:
  ```bash
  git filter-repo --replace-text <(echo 'AIzaSyBsED7R3HF9dGIPQPbQz8F-v3NpkeFPKsc==>REDACTED')
  git push --force-with-lease
  ```
  Skip if the repo is private and only trusted contributors have access.

## 2. Firebase API keys (3 platforms)

**Where they're used:** `lib/firebase_options.dart` (auto-generated by `flutterfire configure`) + `ios/Runner/GoogleService-Info.plist` + `android/app/google-services.json`.

**Important context:** Firebase API keys are *not* secret in the same way Gemini keys are — Google [explicitly documents](https://firebase.google.com/docs/projects/api-keys) that they identify the project but do not authorize access on their own. Access is gated by Firebase Security Rules + App Check. **However**, an unrestricted key still allows attackers to spam costly APIs (Cloud Functions, Storage, Auth) racking up charges.

### Restrictions to apply (do NOT rotate Firebase keys — restrict them)

For each key in https://console.cloud.google.com/apis/credentials :

1. **Application restrictions:**
   - Web key → HTTP referrer restrictions: `your-domain.com/*`, `localhost:*` (for dev).
   - Android key → Package name `com.auracoach.aura_coach_ai` + SHA-1 of release keystore.
   - iOS key → Bundle id `com.auracoach.auraCoachAi`.
2. **API restrictions:**
   - Restrict each key to ONLY the Firebase APIs used:
     - Identity Toolkit API (auth)
     - Cloud Firestore API
     - Firebase Installations API
     - Firebase Cloud Messaging API (if used)
     - Cloud Storage for Firebase API (if used)
3. **Enable App Check** in Firebase Console → Project settings → App Check tab:
   - Register Android with Play Integrity API
   - Register iOS with App Attest
   - Enforce on Firestore + Auth + Functions

### iOS / macOS Firebase project unification

The mismatched iOS project (`emerald-green-1754237937`) means iOS users sign in to a separate Firebase backend than Android/web users — their conversations, library, and sessions don't sync cross-platform.

1. Open Firebase Console → `aura-coach-ai` project → **Add app** → iOS.
2. Bundle id: `com.auracoach.auraCoachAi`.
3. Download the new `GoogleService-Info.plist`.
4. Replace `ios/Runner/GoogleService-Info.plist` with the new file.
5. Run `flutterfire configure --project=aura-coach-ai` from the project root → it will regenerate `lib/firebase_options.dart` with the correct iOS section.
6. Update `ios/Runner/Info.plist` `CFBundleURLSchemes` to the new reverse client ID (look inside the new `GoogleService-Info.plist` for `REVERSED_CLIENT_ID`).
7. Test sign-in on a real device.
8. Delete the old `emerald-green-1754237937` Firebase project once you confirm the unified project works (or keep it empty for billing isolation).

## 3. RevenueCat key

**Where it's used:** `lib/features/subscription/providers/subscription_provider.dart` → `publicApiKey` constant.

**Compromised key:** `test_JHlZAXibjgKpdlshPLWgDXPOSvy`

### Action

The current key is the **sandbox/test** key — RevenueCat scopes test keys to test devices only, so it's safe in the public client. **Before production launch:**

1. Go to https://app.revenuecat.com → your project → API keys.
2. Copy the **production public key** (different from the test key).
3. Replace the constant:
   ```dart
   static const String publicApiKey = 'appl_xxxxxxxxxx'; // iOS
   // or use platform-specific keys:
   //   Platform.isIOS ? 'appl_...' : 'goog_...'
   ```
4. Use the production key in release builds, test key in debug:
   ```dart
   static String get publicApiKey =>
       kDebugMode
           ? 'test_JHlZAXibjgKpdlshPLWgDXPOSvy'
           : (Platform.isIOS ? 'appl_xxx' : 'goog_xxx');
   ```

### Webhook secret

Once Cloud Function webhook is deployed (Subscription Phase F):

1. RevenueCat dashboard → Integrations → Webhooks.
2. URL: your Cloud Function HTTPS endpoint.
3. Authorization header: a long random string you generate.
4. Store the same string in Firebase Functions secret manager:
   ```bash
   firebase functions:secrets:set REVENUECAT_WEBHOOK_AUTH
   ```
5. Cloud Function reads `process.env.REVENUECAT_WEBHOOK_AUTH` and 401s on mismatch.
